# Agent-Issued Mobile Grants Design

## Goal

Let a mobile device connect to a server-agent by scanning or copying a payload
generated by that server-agent, without logging in to Control as a user. The
grant is long-lived and revocable by the server-agent. Control coordinates
pairing, session creation, punch/relay assignment, and observability, but it
does not store any invite secret, grant secret, or other long-lived material
that can be used to access agent services.

## Current State

The current mobile flow requires a Control user token. `POST /sessions` verifies
the user, checks controller registration and device access, selects Relay, signs
session tokens, and adds an `AgentSessionAssignment`. The mobile browser proxy
then opens device-service streams through the same `ControlP2pOrRelayStreamConnector`
as `open_service`.

That remains valid for account-based access. The new flow adds an agent-issued
credential path for mobile SDK/browser-proxy use cases where the user pairs by
QR/copy and should not log in to Control.

## Design Choice

Use short-lived agent invites plus long-lived agent grants. Control does not
verify the grant secret. Instead, Control queues pairing/session requests and the
authenticated server-agent approves or denies them. Control creates the actual
P2P-or-Relay session only after agent approval.

Rejected alternatives:

- Store mobile access grants in Control. This is simple but makes Control the
  long-term access authority, which is the boundary we want to avoid.
- Let Control verify a symmetric grant token. This requires a grant verification
  secret in Control, so Control compromise could authorize mobile access.
- Add a new asymmetric signing dependency immediately. The current workspace
  already has `hmac`, `sha2`, `base64`, and `uuid`, which are sufficient for an
  agent-authoritative first version that does not put grant verification secrets
  in Control.

The selected approach keeps Control in the coordination path and keeps
server-agent as the final authority.

## Credential Types

- `invite`: Short-lived pairing material generated and stored locally by the
  server-agent. It is intended for QR/copy and expires quickly.
- `grant`: Long-lived credential derived from an approved invite and stored by
  the mobile SDK. It contains public metadata plus a `grant_secret`.
- `grant_secret`: Random or derived secret known only to the server-agent and the
  paired mobile device. It is never stored in Control.
- `pairing_proof`: `HMAC-SHA256(invite_secret, canonical_pairing_request)`.
- `session_proof`: `HMAC-SHA256(grant_secret, canonical_session_request)`.

## Invite Payload

The server-agent SDK exposes a function that creates an invite and returns a
copy/QR-safe payload:

```json
{
  "version": 1,
  "control_url": "https://control.example.com",
  "device_id": "pc_001",
  "invite_id": "inv_...",
  "invite_secret": "base64url-128-bit-secret",
  "agent_p2p_cert_fingerprint": "sha256:...",
  "allowed_services": ["svc_web_3000"],
  "expires_at": 1767000000,
  "max_uses": 1
}
```

Agent local state stores the invite as:

```json
{
  "invite_id": "inv_...",
  "invite_secret": "server-local-secret",
  "allowed_services": ["svc_web_3000"],
  "expires_at": 1767000000,
  "max_uses": 1,
  "uses": 0,
  "revoked": false
}
```

The invite secret is short-lived. Control never stores it.

## Pairing Flow

1. Server-agent creates an invite locally and displays the payload as text/QR.
2. Mobile SDK imports the payload and chooses a `client_id`.
3. Mobile sends a pairing request to Control without a user token:

```json
{
  "device_id": "pc_001",
  "invite_id": "inv_...",
  "client_id": "mobile_001",
  "requested_services": ["svc_web_3000"],
  "nonce": "base64url-random",
  "proof": "hmac-sha256(invite_secret, canonical_pairing_request)"
}
```

4. Control stores a pending pairing request. It stores routing metadata and the
   proof, not `invite_secret`.
5. The authenticated server-agent polls Control for pending pairing requests for
   its `device_id`.
6. Server-agent verifies:
   - invite exists locally
   - invite is not expired, revoked, or over `max_uses`
   - requested services are a subset of invite scope
   - proof matches the canonical request and invite secret
7. Server-agent creates a long-lived grant record locally and approves the
   pending request through Control.
8. Mobile polls Control for the pairing result. On approval it receives public
   grant metadata. Mobile derives or stores `grant_secret` locally using the
   invite material and approved grant id.

Control sees the HMAC proof but cannot derive `invite_secret` or `grant_secret`.
Replay is blocked by agent-side invite use counters and nonces.

## Grant Record

Agent-local grant record:

```json
{
  "grant_id": "gr_...",
  "client_id": "mobile_001",
  "allowed_services": ["svc_web_3000"],
  "grant_secret": "server-local-secret",
  "enabled": true,
  "revocation_version": 1,
  "created_at": 1760000000,
  "last_used_at": null
}
```

Mobile grant credential:

```json
{
  "version": 1,
  "control_url": "https://control.example.com",
  "device_id": "pc_001",
  "grant_id": "gr_...",
  "client_id": "mobile_001",
  "allowed_services": ["svc_web_3000"],
  "grant_secret": "mobile-local-secret",
  "revocation_version": 1
}
```

The mobile credential is sensitive and must be stored in the SDK credential
store. Copying the public grant id is not enough to access services; each
session requires a proof over the request using `grant_secret`.

## Grant-Based Session Flow

Mobile does not call the current user-token `POST /sessions` path. It uses a new
grant-session flow:

1. Mobile sends `POST /agent-grant/sessions/start` without a user token:

```json
{
  "client_id": "mobile_001",
  "device_id": "pc_001",
  "service_id": "svc_web_3000",
  "grant_id": "gr_...",
  "revocation_version": 1,
  "nonce": "base64url-random",
  "proof": "hmac-sha256(grant_secret, canonical_session_request)"
}
```

2. Control stores a pending grant-session request and returns:

```json
{
  "pending_session_id": "gps_...",
  "poll_interval_ms": 1000,
  "expires_at": 1767000000
}
```

3. Authenticated server-agent polls pending grant-session requests.
4. Server-agent verifies:
   - grant exists locally
   - grant is enabled
   - request `revocation_version` matches the local grant
   - requested `service_id` is in grant scope
   - session proof matches the canonical request and grant secret
5. Server-agent approves or denies the pending request through Control.
6. On approval, Control creates the normal session assignment, selects Relay,
   issues short-lived session access/relay tokens, and stores grant metadata on
   the `AgentSessionAssignment`.
7. Mobile polls `GET /agent-grant/sessions/{pending_session_id}` until it
   receives the same `CreateSessionResponse` shape used by the normal path.
8. Browser proxy opens the device-service stream over P2P first and Relay only
   after fallback.

Control does not issue a session token before agent approval.

## Agent Claim Enforcement

The approved `AgentSessionAssignment` includes non-secret grant metadata:

```json
{
  "grant_id": "gr_...",
  "grant_revocation_version": 1,
  "grant_service_id": "svc_web_3000"
}
```

When agent claims the session, it verifies that the local grant still exists,
is enabled, has the same revocation version, and allows the service. If the
grant was revoked after approval but before claim, the agent refuses the
session. This is the final enforcement layer.

## Browser Proxy Data Flow

After the grant-based tunnel starts, the browser proxy flow is unchanged:

```text
WebView/program
  -> local HTTP proxy on mobile
  -> synthetic device-service host
  -> mobile SDK starts grant-authorized session through Control
  -> P2P QUIC stream to agent
  -> Relay fallback if P2P fails
  -> agent validates grant and target policy
  -> registered service target
```

The proxy does not know invite internals. It only uses a tunnel client that has
a valid grant credential.

## Revocation

Revocation is long-term and agent-authoritative:

- Agent can disable a grant.
- Agent can increment `revocation_version`.
- New grant-session requests with the old version are denied.
- Already approved but unclaimed sessions are rejected at claim time.
- The first implementation does not forcibly close already established streams
  when a grant is revoked. Revocation blocks new grant-session approvals and
  blocks approved-but-unclaimed sessions at agent claim time.

## Security Properties

- Control compromise does not reveal invite secrets or grant secrets.
- Control cannot create a usable service session without agent approval.
- A copied invite is only useful before expiry and before `max_uses` is reached.
- A copied grant id is not sufficient without the grant secret.
- Agent revocation is authoritative even if Control has stale metadata.
- Relay tokens remain short-lived per-session credentials, not long-lived mobile
  authorization.

## First Implementation Scope

Implement the smallest useful version:

1. Shared protocol/control-client data types for invite payloads, pairing
   requests/responses, grant credentials, and grant-session requests.
2. Control endpoints:
   - submit pairing request without user token
   - agent lists and approves/denies pending pairing requests using agent token
   - submit grant-session request without user token
   - agent lists and approves/denies pending grant-session requests
   - mobile polls pairing/session result
3. Agent-side SDK/runtime helpers:
   - create invite
   - approve pairing request
   - store/revoke/list grants in memory first, then file-backed storage
   - verify grant metadata before accepting claimed sessions
4. Mobile SDK helpers:
   - import invite payload
   - complete pairing through Control
   - start P2P-or-Relay tunnel using stored grant instead of user token
5. CLI/documentation support for manual smoke:
   - agent prints invite payload
   - mobile imports invite payload and starts browser proxy

## Out Of Scope For First Implementation

- Replacing the current user-token flow. Both flows should coexist.
- Cloud push notification delivery for pairing requests.
- Closing already active streams immediately when a grant is revoked.
- Adding asymmetric signatures. The async agent-approval flow avoids storing
  grant verification secrets in Control for the first implementation.
